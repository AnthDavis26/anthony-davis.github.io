June 19, 2024

Well, so much for weekly updates. It's been very busy in my personal life.

In terms of my projects, most of my time has been spent on Post-Apocalyptic George. I have further condensed my code to be easier to manage. Before now, the logic to dictate what action the player takes based on their input consisted largely of switch-case blocks;

Example

"PromptUntilConditionMet(PressedOne, PressedTwo, etc);
Case (choice=='1') ....Do This; break;
Case (choice=='2') ....Do That; break;
Case (choice=='3') ....Do Something Else; break;
Case (choice=='p') ....Pause Game; break;
Case (choice=='t') ....Enter Text; break;
Case (other) ...Etc..."

That's pretty cumbersome. This sort of logic would be carried over to countless functions in the game. It's too difficult to manage, and thus more prone to error, and ultimately harder to debug. It's simply time-consuming.

Presumably, the player will be able to type 'p' or 't' at most points in the game. So firstly, we can condense this part into a PromptUntil sort of function and to avoid accidentally forgetting to check for those commands.

"PromptUntilConditionMet(PressedOne, PressedTwo, etc);
Case (choice=='1') ....Do This; break;
Case (choice=='2') ....Do That; break;
Case (choice=='3') ....Do Something Else; break;
Case (other) ...Etc..."

I opted to create a function that allows these sorts of arguments instead:

"PromptUntilCommand(
....('1', DoThis),
....('2', DoThat),
....('3', SomethingElse),
...etc
)"

No bulky switch-case blocks, simplified logic, and nicer to look at.

It's a small update in terms of concept, but the implementation was a learning curve. I'm still getting used to C++ variadic templates, including code to accept multiple functions as parameters (as seen for PromptUntilCommand()).

Post-Apocalyptic George (C++ Text-Based Game)

Download v0.0.1 Here

While my overall plan for the rest of the game is vague at best, I felt it was time for me to incorporate how the player can make decisions. As this game is intended to be dry and comical with bizarre elements, I re-incorporated a feature that I had in this project a few years ago…

Making George Eat Trash

My wife was adamant that I bring back this ability, and I was inclined to oblige.

For the most part, I imagine that player decisions will be multiple-choice. But I want the player to also be able to think outside the box and be rewarded for it, or at least amused by the result. This is why custom prompt inputs exist in the game.

Each word from the user's input is compared to a custom thesaurus (the words to compare vary depending on context within the game). Once corresponding synonyms for both “eat” and “trash” are found, the input is deemed correct to make George eat trash. Excellent.

The thesaurus stores unordered sets of synonyms for different words, so searching for matching synonyms has O(1) complexity (it does not depend on the length of the user's input string). The primary operation is to scan each character in the user input to form a temporary substring to check for a synonym. Thus the primary operation has a time complexity of O(n); the number of steps taken to finish the process, in the worst cases, steadily increases the more characters there are to scan.

I would love to be able to incorporate AI to parse user input in a more meaningful way, but I don't have that kind of knowledge. Maybe in the future.
